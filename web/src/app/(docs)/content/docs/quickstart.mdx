Kizlo's surface and how its pieces fit together.

<Callout>

## Server & browser clients

Installation gave you two clients that behave the same way — the only difference is where they run:

- **Server client** — imported from `@/lib/kizlo/server`, for server-to-server calls.
- **Browser client** — imported from `@/lib/kizlo/client`, for browser-to-server calls.

Both share the same namespaces and call style (`client.posts.list(...)`). The only difference is **scope**:
a procedure marked `internal` is dropped from the browser client and runs on the server only — everything
else is available on both. The SEO procedures are a good example:

```ts title="src/app/sitemap.ts"
import { client } from "@/lib/kizlo/server"

// ✅ internal procedures like `seo` are available on the server client
const { data } = await client.seo.sitemaps()
```

```ts title="src/app/seo-widget.tsx"
import { client } from "@/lib/kizlo/client"

// ❌ `seo` is omitted from the browser client — not on the namespace,
// and rejected at runtime if you reach for it anyway
client.seo.sitemaps()
```

<Callout>
  This page uses `client` generically; reach for whichever fits where your code runs.
</Callout>

## Core APIs

Out of the box, the `client` exposes standard WordPress content as procedures. Every call returns a
`{ data, error }` result you check directly, and also has a `.call()` variant that returns the data and
throws a `KizloError` instead of returning it.

```ts
// Posts
const { data } = await client.posts.list({ query: { perPage: 10 } })
const post = await client.posts.get.call({ params: { identifier: "hello-world" } })

// Comments
await client.comments.submit({ body: { post: 42, content: "Great read!" } })

// Menus
const menus = await client.menus.list()
```

## Plugins & extensions

Anything beyond core WordPress comes from an **extension** — an installable npm package that adds its own
namespace to the client. An extension can be standalone, but many bridge a WordPress plugin, and that bridge
has two sides: the npm package in your app, and a matching **Kizlo companion plugin** in WordPress that makes
the third-party plugin reachable.

What the companion plugin does varies by integration:

- **Kizlo for WooCommerce** glues WooCommerce in and layers extra features on top.
- **Kizlo for Contact Form 7** simply exposes CF7's API so the extension can call it.

Either way, you install both the third-party plugin and its Kizlo companion in WordPress, then add the npm
package to your app:

| npm package | WordPress plugins | Adds |
| --- | --- | --- |
| `@kizlo/woocommerce` | WooCommerce + Kizlo for WooCommerce | `client.woocommerce` — products, cart, checkout, customers |
| `@kizlo/cf7` | Contact Form 7 + Kizlo for Contact Form 7 | `client.contactForm` — form submissions |

Register the extension on your server — both WordPress plugins must already be active:

```ts title="src/lib/kizlo/server/index.ts"
import { createKizlo } from "kizlo/nextjs/server"
import { woocommerce } from "@kizlo/woocommerce"

export const { client, handler } = createKizlo({
  extensions: [woocommerce()],
})
```

Now the extension's API is available on the client, exactly like the core ones:

```ts
const { data } = await client.woocommerce.products.list({ query: { perPage: 12 } })
```

## Custom extensions

When no package fits, you can author your own extension and get a first-class API layer of your own. Inside
a procedure you also have the **WordPress client** at `context.service.wordpress`, so you can build on top
of the REST API — for example, wrapping the core post endpoint and returning a shape that's yours, not
WordPress's:

```ts title="src/lib/kizlo/server/featured.ts"
import { createExtension, createProcedure } from "kizlo"
import { z } from "zod"

export const featured = createExtension({
  id: "featured",
  init: () => ({
    router: {
      get: createProcedure(
        {
          scope: "api",
          method: "GET",
          path: "/featured/{id}",
          params: z.object({ id: z.coerce.number() }),
          output: z.object({ id: z.number(), headline: z.string() }),
        },
        async ({ input, context }) => {
          const { data, error } = await context.service.wordpress.posts.get({ id: input.params.id })
          if (error) throw error

          return { id: data.id, headline: data.title.rendered }
        },
      ),
    },
  }),
})
```

Register it the same way you'd register any extension, and `client.featured.get(...)` is driven by your
`output` — change the shape and every call site updates with it.

## Adapters

Adapters let you plug Kizlo into your own infrastructure — logging, auth, captcha, geo — with no lock-in.
For example, wire up logging with the built-in console adapter (or pass your own function):

```ts title="src/lib/kizlo/server/index.ts"
import { createKizlo } from "kizlo/nextjs/server"
import { consoleLog } from "kizlo"

export const { client, handler } = createKizlo({
  adapters: {
    logger: consoleLog(),
  },
})
```

<Callout>
  You're not limited to the adapters Kizlo ships — write your own to fit Kizlo into whatever service or
  infrastructure you already use.
</Callout>
