Kizlo talks to WordPress. You can point it at your own, or let Kizlo run one
locally in Docker for development. Pick the one that fits — both end at the same
[quickstart](/docs/quickstarts/nextjs).

    ### Run WordPress locally

    For local development you don't need your own WordPress or a plugin download. During the quickstart,
    `npx kizlo dev` boots a self-contained WordPress in Docker with the Kizlo plugin already
    bundled, provisions an Application Password, and writes it to `.env` — so the local stack is
    ready to call with nothing to set up by hand.

    The one thing to have ready now is [Docker Desktop](https://www.docker.com/products/docker-desktop/),
    installed and running.

    <Callout>
      That's the whole setup for local dev. Head to the [quickstart](/docs/quickstarts/nextjs) —
      `kizlo dev` handles the plugin and credentials for you.
    </Callout>
  </TabsContent>

  <TabsContent value="site">
    ### Connect your own WordPress

    Pointing Kizlo at a live site or an existing WordPress install takes two steps.

    <Steps>

    <Step>

    #### Install & activate the WordPress plugin

    Download the latest Kizlo plugin, then in WP Admin go to **Plugins → Add New → Upload Plugin**,
    choose the zip, and **Activate**.

    <DownloadButton plugin="kizlo" />

    Once activated, the plugin keeps itself up to date from within WordPress — no need to come back here for future versions.

    </Step>

    <Step>

    #### Create an admin Application Password

    In WP Admin go to **Users → Profile → Application Passwords** (on a user with admin capability),
    give it a name like `kizlo`, and generate it.

    This is the credential Kizlo uses to talk to WordPress. Copy it now — WordPress only shows it
    once — and keep it for the quickstart.

    </Step>

    </Steps>
  </TabsContent>
</Tabs>

With WordPress ready either way, head to the [Next.js quickstart](/docs/quickstarts/nextjs) to add
Kizlo to your app and make your first call.
